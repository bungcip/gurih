## 2024-05-03 - [Fix] Add rate limiting tracking for long passwords to prevent bypass
**Vulnerability:** An attacker could bypass rate limits by sending excessively long passwords, which would correctly trigger an early rejection with a dummy hash (to avoid timing attacks and CPU exhaustion) but the attempt was NOT tracked in `login_attempts`. This converted an authentication bypass attempt into a CPU exhaustion DoS by skipping the IP ban mechanism.
**Learning:** Early return paths for security rejection (like input length limits) MUST still integrate with secondary security systems like rate-limiting, otherwise attackers can spam the "safe" rejection path to consume server resources (e.g. PBKDF2 dummy hashing).
**Prevention:** Ensure that invalid inputs are tracked by rate limiters unless tracking them introduces a worse vulnerability (e.g., ignoring overly long usernames to prevent Out-Of-Memory caching attacks).
